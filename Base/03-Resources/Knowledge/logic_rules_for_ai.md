## 📋 КАТАЛОГ ПРАВИЛ

### Раздел 1: Работа с файлами

#### Правило 1: Переименование файла

**ЕСЛИ:** Файл переименован  
**ТО:** 
1. Найти все ссылки на старый файл (grep_search)
2. Исправить ссылки в каждом файле (edit)
3. Проверить что ссылок на старый файл не осталось
4. Закоммитить изменения

**ПОЧЕМУ:** Предотвратить битые ссылки

**Пример:**
```
ЕСЛИ: CHEAT_SHEET.md → 02-Areas/Documentation/CHEAT_SHEET.md
ТО: 
  1. grep_search -pattern "CHEAT_SHEET.md"
  2. Найти 9 ссылок
  3. Исправить: ./CHEAT_SHEET.md → ../02-Areas/Documentation/CHEAT_SHEET.md
  4. git commit -m "Fix: ссылки после перемещения"
```

---

#### Правило 2: Удаление файла

**ЕСЛИ:** Файл удалён  
**ТО:**
1. Создать бэкап (если ценный)
2. Найти все ссылки на файл
3. Исправить или удалить ссылки
4. Закоммитить удаление

**ПОЧЕМУ:** Сохранить историю, предотвратить ошибки

**Пример:**
```
ЕСЛИ: TEMP/ папка удалена
ТО:
  1. pre-operation-backup.ps1 (бэкап)
  2. grep_search -pattern "TEMP/"
  3. Исправить ссылки
  4. git commit -m "Cleanup: TEMP удалена"
```

---

#### Правило 3: Создание нового файла

**ЕСЛИ:** Создан новый файл  
**ТО:**
1. Определить уровень правила (1-4)
2. Обновить ALL_RULES_CATALOG.md
3. Добавить в RULES_INDEX.md
4. Создать тест-кейсы (если правило)
5. Закоммитить

**ПОЧЕМУ:** Поддерживать актуальность каталогов

**Пример:**
```
ЕСЛИ: 03-Resources/PowerShell/new-script.ps1 создан
ТО:
  1. Уровень 3 (скрипт)
  2. ALL_RULES_CATALOG.md → добавить в каталог
  3. RULES_INDEX.md → добавить в индекс
  4. RULE_TEST_CASES.md → добавить тест
  5. git commit -m "Add: new-script.ps1"
```

---

### Раздел 2: Тестирование

#### Правило 4: Запуск тестов

**ЕСЛИ:** Изменения внесены  
**ТО:**
1. Создать _TEST_ENV (если нет)
2. Скопировать изменения
3. Запустить test-all-changes.ps1
4. Если 100% → закоммитить
5. Если <100% → исправить ошибки

**ПОЧЕМУ:** Гарантировать стабильность

**Пример:**
```
ЕСЛИ: check-rule-profiles.ps1 изменён
ТО:
  1. create-test-env.ps1 -Force
  2. Copy-Item check-rule-profiles.ps1
  3. test-all-changes.ps1
  4. Результат: 100%
  5. git commit
```

---

#### Правило 5: Провал тестов

**ЕСЛИ:** Тесты провалены (<100%)  
**ТО:**
1. Прочитать отчёт (COMPREHENSIVE_TEST_REPORT.md)
2. Найти проваленные тесты
3. Исправить причину
4. Перезапустить тесты
5. Повторять пока 100%

**ПОЧЕМУ:** Не коммитить с ошибками

**Пример:**
```
ЕСЛИ: 24/25 тестов (96%)
ТО:
  1. Отчёт: "Матрица профилей" провалено
  2. Причина: Нет слова "МАТРИЦА" в RULES_INDEX.md
  3. Исправление: Изменить тест на "Содержит индекс"
  4. Перезапуск: 25/25 (100%)
  5. git commit
```

---

### Раздел 3: Бэкап

#### Правило 6: Бэкап перед операцией

**ЕСЛИ:** Планируется опасная операция  
**ТО:**
1. Запустить pre-operation-backup.ps1
2. Указать тип операции
3. Проверить что бэкап создан
4. Только потом выполнять операцию

**ПОЧЕМУ:** Возможность отката

**Пример:**
```
ЕСЛИ: Удаление TEMP (~500 MB)
ТО:
  1. pre-operation-backup.ps1 -OperationType "Cleanup"
  2. Бэкап: D:\QwenPoekt\_BACKUP\TEMP_Full_Backup
  3. Проверка: 683 файла скопировано
  4. Удаление: Remove-Item TEMP -Recurse
```

---

#### Правило 7: Восстановление из бэкапа

**ЕСЛИ:** Нужен откат изменений  
**ТО:**
1. Найти бэкап в _BACKUP/
2. Проверить целостность
3. Восстановить файлы
4. Закоммитить восстановление

**ПОЧЕМУ:** Быстрое восстановление

### Раздел 4: Коммиты

#### Правило 8: Когда коммитить

**ЕСЛИ:** Завершён крупный этап  
**ИЛИ:** Добавлен новый файл  
**ИЛИ:** Исправлена критичная ошибка  
**ТО:** Закоммитить

**ПОЧЕМУ:** Сохранить прогресс

**Пример:**
```
ЕСЛИ: 25/25 тестов пройдено
ИЛИ: Создан RULES_GRAPH.md
ИЛИ: Исправлено 100% ошибок
ТО: git add . && git commit -m "..."
```

---

#### Правило 9: Когда НЕ коммитить

**ЕСЛИ:** Косметика (опечатка)  
**ИЛИ:** Дополнение (абзац)  
**ИЛИ:** Промежуточное изменение  
**ТО:** Не коммитить, ждать завершения

**ПОЧЕМУ:** Избегать мусорных коммитов

---

#### Правило 10: Форматирование коммита

**ЕСЛИ:** Коммит готовится  
**ТО:** Использовать формат:

```
Type: Описание

Details (опционально)
```

**Типы:**
- `Add:` — Новый файл
- `Update:` — Обновление
- `Fix:` — Исправление ошибки
- `Cleanup:` — Очистка
- `Refactor:` — Рефакторинг

**Пример:**
```
Add: RULES_GRAPH.md — граф связей правил

Создан граф причинно-следственных связей
между правилами Базы Знаний.
```

---

### Раздел 5: Ошибки

#### Правило 11: Обнаружение ошибки

**ЕСЛИ:** Обнаружена ошибка  
**ТО:**
1. Записать в OPERATION_LOG.md
2. Определить критичность
3. Если критичная → исправить немедленно
4. Если некритичная → записать в задачи

**ПОЧЕМУ:** Не терять ошибки

---

#### Правило 12: Исправление ошибки

**ЕСЛИ:** Ошибка исправлена  
**ТО:**
1. Записать решение в error_solutions.md
2. Обновить OPERATION_LOG.md
3. Закоммитить исправление

**ПОЧЕМУ:** Сохранить знание для будущего

---

## 🔄 РАЗДЕЛ 7: НЕМОНOTОННАЯ ЛОГИКА (ПРАВИЛА С ИСКЛЮЧЕНИЯМИ)

### 7.1. Правила по умолчанию

**Определение:** Правила, которые действуют, если нет противоречащих фактов.

**Формат:**
```markdown
**Правило:** [Текст правила]

**По умолчанию:** [Действие по умолчанию]

**Исключения:**
- ЕСЛИ [условие 1] → [исключение 1]
- ЕСЛИ [условие 2] → [исключение 2]
```

---

### Правило 16: Именование файлов (с исключениями)

**Правило:** Все файлы именуются snake_case

**По умолчанию:** Использовать snake_case

**Исключения:**
- ЕСЛИ файл — бэкап → Допустим `Backup_2026-03-02`
- ЕСЛИ файл — черновик → Допустим `WIP_название.md`
- ЕСЛИ файл — тестовый отчёт → Допустим `TEST_REPORT_ГГГГ-ММ-ДД.md`

**Пример:**
```
Правило: snake_case для всех файлов

Исключения:
  ✅ Backup_2026-03-02 (бэкап)
  ✅ WIP_new_feature.md (черновик)
  ✅ TEST_REPORT_2026-03-02.md (отчёт)
```

---

### Правило 17: Коммиты (с исключениями)

**Правило:** Коммитить только крупные этапы

**По умолчанию:** Ждать завершения крупного этапа

**Исключения:**
- ЕСЛИ исправлена критичная ошибка → Коммитить немедленно
- ЕСЛИ создан новый файл → Коммитить сразу
- ЕСЛИ день заканчивается → Коммитить прогресс (auto-commit-daily.ps1)

**Пример:**
```
Правило: Коммитить крупные этапы

Исключения:
  ✅ Критичная ошибка исправлена → Коммит
  ✅ Новый файл создан → Коммит
  ✅ 18:00 (ежедневный коммит) → Коммит
```

---

### Правило 18: Тестирование (с исключениями)

**Правило:** Всегда тестировать в _TEST_ENV перед коммитом

**По умолчанию:** Копировать в _TEST_ENV → Тесты → Коммит

**Исключения:**
- ЕСЛИ косметика (опечатка) → Коммитить без тестов
- ЕСЛИ срочное исправление → Коммитить → Тестировать постфактум
- ЕСЛИ тестовая среда недоступна → Задокументировать → Коммитить

**Пример:**
```
Правило: Тестировать в _TEST_ENV

Исключения:
  ✅ Опечатка исправлена → Коммит без тестов
  ✅ Критичная ошибка → Коммит → Тесты
  ✅ _TEST_ENV недоступна → Документ → Коммит
```

### 7.2. Приоритет правил

**Когда правила конфликтуют:**

1. **Специфичное правило > Общее правило**
2. **Исключение > Правило по умолчанию**
3. **Красный уровень > Жёлтый > Зелёный**

**Пример:**
```
Общее: "Всегда тестировать"
Специфичное: "Опечатки без тестов"

Применяется: Специфичное (опечатка → без тестов)
```

---

### 7.3. Отрицание как неудача

**Определение:** Если не удалось доказать истинность → считаем ложным.

**Применение:**
```
ЕСЛИ не удалось доказать "файл существует" → Файл не существует
ЕСЛИ не удалось доказать "ссылка рабочая" → Ссылка битая
ЕСЛИ не удалось доказать "тесты пройдены" → Тесты провалены
```

**Пример:**
```powershell
# Проверка файла
if (!(Test-Path $file)) {
    # Файл не существует (отрицание как неудача)
    Write-Error "Файл не найден"
}
```

---

### 7.4. Закрытие мира (Closed World Assumption)

**Определение:** Всё, что не известно как истинное → считается ложным.

**Применение в базе знаний:**
```
ЕСЛИ правило не записано в базе → Правило не существует
ЕСЛИ связь не указана в графе → Связи нет
ЕСЛИ тест не прошёл → Тест провален (не "неизвестно")
```

**Пример:**
```
Вопрос: "Можно ли коммитить без тестов?"
База знаний: Нет правила "коммитить без тестов"
Вывод: Нельзя (закрытие мира)
```

## 🕐 РАЗДЕЛ 8: ВРЕМЕННЫЕ ПРАВИЛА (ТЕМПОРАЛЬНАЯ ЛОГИКА)

### 8.1. Временные операторы

**Определение:** Правила с временной привязкой.

**Операторы:**
- **ВСЕГДА (always)** — Действует всегда
- **СНАЧАЛА (before)** — Должно произойти раньше
- **ЗАТЕМ (then)** — Должно произойти позже
- **ПОСЛЕ (after)** — Действует после события
- **НИКОГДА (never)** — Никогда не должно произойти

---

### Правило 19: Бэкап (временное правило)

**Правило:** ВСЕГДА (сначала бэкап, затем операция)

**Временная последовательность:**
```
1. СНАЧАЛА: pre-operation-backup.ps1
2. ЗАТЕМ: Операция (удаление, изменение)
3. ПОСЛЕ: Проверка целостности
```

**Формат:**
```
ВСЕГДА (
  ЕСЛИ операция →
    СНАЧАЛА бэкап →
    ЗАТЕМ операция →
    ПОСЛЕ проверка
)
```

**Пример:**
```powershell
# Удаление TEMP
.\scripts\pre-operation-backup.ps1 -OperationType "Cleanup"  # СНАЧАЛА
Remove-Item TEMP -Recurse                                     # ЗАТЕМ
.\scripts\check-integrity.ps1                                 # ПОСЛЕ
```

---

### Правило 20: Тестирование (временное правило)

**Правило:** ВСЕГДА (сначала тесты, затем коммит)

**Временная последовательность:**
```
1. СНАЧАЛА: test-all-changes.ps1
2. СНАЧАЛА: 100% тестов пройдено
3. ЗАТЕМ: git add .
4. ЗАТЕМ: git commit -m "..."
```

**Исключения:**
- ЕСЛИ косметика → Пропустить тесты → Коммит
- ЕСЛИ срочно → Коммит → ЗАТЕМ тесты

**Формат:**
```
ВСЕГДА (
  ЕСЛИ коммит →
    СНАЧАЛА тесты (100%) →
    ЗАТЕМ коммит
)
```

---

### Правило 21: Архивация (временное правило)

**Правило:** ВСЕГДА (после 90 дней → архив)

**Временное условие:**
```
ЕСЛИ last_reviewed < (сегодня - 90 дней) →
  ЗАТЕМ предупредить (⚠️ Устаревает) →
  ПОСЛЕ 7 дней →
    ЗАТЕМ архивировать (_archive/)
```

**Пример:**
```powershell
# Проверка актуальности
.\scripts\check-rules-freshness.ps1 -MaxAge 90

# Если устаревшее → предупредить
# Если 7 дней прошло → архивировать
```

### 8.2. Временные цепочки

**Определение:** Последовательность событий во времени.

**Пример цепочки:**
```
Создание файла (_drafts/)
    ↓ (затем)
Тестирование (_TEST_ENV)
    ↓ (затем)
100% тестов пройдено
    ↓ (затем)
Внедрение (Base/)
    ↓ (затем)
Коммит
    ↓ (после)
Очистка (_drafts/, _TEST_ENV)
```

**Формат:**
```
ВСЕГДА (
  СНАЧАЛА _drafts/ →
  ЗАТЕМ _TEST_ENV →
  ЗАТЕМ 100% →
  ЗАТЕМ Base/ →
  ЗАТЕМ коммит →
  ПОСЛЕ очистка
)
```

## 🤖 РАЗДЕЛ 10: НЕЙРОСИМВОЛИЧЕСКИЙ ПОДХОД

### 10.1. LLM как преобразователь запросов

**Определение:** LLM преобразует естественный язык в логический запрос.

**Процесс:**
```
1. Пользователь задаёт вопрос на естественном языке
2. LLM анализирует вопрос
3. LLM генерирует логический запрос (функция из AI_FUNCTIONS.md)
4. Логический движок выполняет запрос
5. LLM формулирует ответ на естественном языке
```

**Пример:**
```
Вопрос: "Почему тесты провалены?"

LLM анализ:
  - Тип вопроса: причинность
  - Нужная функция: get_causal_chain

LLM генерация:
  get_causal_chain(effect: "тесты провалены")

Логический вывод:
  chain: [
    {"cause": "скрипты используют Base", "effect": "тесты провалены"},
    {"cause": "пути неверные", "effect": "скрипты используют Base"}
  ]

LLM ответ:
  "Тесты провалены из-за неправильных путей в скриптах.
   Причинно-следственная цепочка:
   1. Скрипты используют пути D:\\QwenPoekt\\Base
   2. Тесты запускаются в _TEST_ENV
   3. Пути не совпадают → тесты провалены"
```

---

### 10.2. LLM как генератор гипотез

**Определение:** LLM предлагает правила/гипотезы для проверки.

**Процесс:**
```
1. LLM анализирует базу знаний
2. LLM предлагает гипотезу (новое правило)
3. Логический движок проверяет на непротиворечивость
4. Если OK → Предложить Пользователю
5. Если утверждено → Добавить в базу
```

**Пример:**
```
LLM анализ: "Замечено: все правила имеют last_reviewed"

LLM гипотеза:
  "Правило: Все правила должны иметь last_reviewed"

Проверка:
  check_consistency() → ✅ Нет противоречий

Предложение Пользователю:
  "Добавить правило: 'Все правила должны иметь last_reviewed'?"

Если утверждено → Добавить в logic_rules_for_ai.md
```

### 10.4. Гибридные RAG-системы


**Пример:**
```

Логический вывод:
  - check_rule_relations("тестирование") → 5 правил

LLM ответ:
  "К тестированию относятся 5 правил:
   1. test-all-changes.ps1
   2. RULE_TEST_CASES.md
   3. _TEST_ENV
   4. ...
   Эти правила связаны отношением 'requires'."
```
