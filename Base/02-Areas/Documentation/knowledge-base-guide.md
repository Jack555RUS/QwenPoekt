## 1. Введение: Что такое База Знаний (Ядро)?

**База Знаний (Knowledge Base, Ядро)** — это централизованное, структурированное хранилище всей информации для разработки Unity проекта и обучения ИИ-ассистента.

**Включает:**

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Знания** | Статьи, документация, заметки | `KNOWLEDGE_BASE/` (740 файлов) |
| **Код** | Скрипты, утилиты, пайплайны | `03-Resources/PowerShell/` (40+ файлов) |
| **Шаблоны** | Готовые решения | `_templates/` |
| **Отчёты** | История операций | `reports/` |
| **Метаданные** | Описания, версии | Оглавления файлов |

**Назначение:**

- ✅ Избегать дублирования знаний
- ✅ Обеспечивать воспроизводимость
- ✅ Ускорять разработку
- ✅ Сохранять корпоративную память

## 2. Принципы накопления Базы Знаний

### 2.1. Источники пополнения

| Источник | Что добавляем |
|----------|---------------|
| **Внешние** | Документация Unity, статьи, туториалы |
| **Внутренние** | Результаты экспериментов, решения ошибок |
| **Код** | Скрипты автоматизации, утилиты |
| **Метаданные** | Логи сессий, отчёты, заметки |

### 2.2. Критерии отбора (что добавлять)

**Добавляем, если:**

- ✅ Потенциально переиспользуемо
- ✅ Имеет ценность для будущих проектов
- ✅ Может быть задокументировано
- ✅ Не является временным черновиком

**Не добавляем:**

- ❌ Сырые результаты одного эксперимента
- ❌ Черновики без завершения
- ❌ Дубликаты существующих знаний

### 2.3. Процесс добавления

```
1. Загрузка/создание материала
   ↓
2. Проверка целостности
   ↓
3. Документирование (оглавление, метаданные)
   ↓
4. Размещение в соответствующем разделе
   ↓
5. Обновление каталога
```

## 3. Анализ поступающей информации

### 3.1. Классификация

| Категория | Критерии | Куда сохранять |
|-----------|----------|----------------|
| **Правила** | Стандарты, best practices | `01_RULES/` |
| **Паттерны** | Решения ошибок, фичи | `03_PATTERNS/` |
| **Инструменты** | Скрипты, утилиты | `03-Resources/PowerShell/` |
| **Фундамент** | Основы, глоссарии | `00_CORE/` |
| **Методологии** | Процессы, циклы | `05_METHODOLOGY/` |

### 3.2. Процесс анализа статьи/документа

```
1. Быстрое сканирование (заголовок, выводы)
   ↓
2. Выделение ключевых идей
   ↓
3. Оценка применимости к проекту
   ↓
4. Извлечение сути (резюме)
   ↓
5. Связывание с существующими знаниями
   ↓
6. Решение: сохранить (полный текст или выжимку)
```

### 3.3. Логика принятия решений

**Вопросы перед добавлением:**

```
[ ] 1. Уникальность: есть ли уже такое в Базе?
[ ] 2. Качество: насколько информация полна?
[ ] 3. Применимость: будет ли востребована?
[ ] 4. Издержки: размер, сложность обработки?
[ ] 5. Юридическая чистота: можно ли хранить?
```

## 5. Как пользоваться Базой Знаний

### 5.1. Поиск и извлечение

**Инструменты:**

```powershell
# Поиск по содержимому
grep_search -pattern "UI Toolkit" -glob "*.md"

# Поиск по имени файла
glob -pattern "**/ui_*.md"

# Поиск в конкретной папке
grep_search -pattern "button" -path "KNOWLEDGE_BASE/01_RULES"
```

**Каталоги:**

- [`KNOWLEDGE_BASE/00_README.md`](KNOWLEDGE_BASE/00_README.md) — навигатор по знаниям
- [`AI_START_HERE.md`](AI_START_HERE.md) — главная инструкция
- [`README.md`](README.md) — навигатор по проекту

### 5.2. Интеграция в проекты

**Относительные пути:**

```csharp
// В коде Unity
string knowledgePath = Path.Combine(Application.dataPath, "../Base/KNOWLEDGE_BASE");
```

**Символические ссылки (опционально):**

```powershell
# Создать ссылку в проекте
New-Item -Path "PROJECTS/DragRaceUnity/Docs" -ItemType SymbolicLink -Value "D:\QwenPoekt\Base\KNOWLEDGE_BASE"
```

### 5.3. Версионирование

**Git для кода и знаний:**

```powershell
# Коммит важных изменений
git add 03-Resources/Knowledge/01_RULES/new_rule.md
git commit -m "Add: новое правило именование"
```

**Локальные метки:**

- ✅ **Актуально** — используется сейчас
- ⚠️ **Устаревает** — скоро будет заменено
- ❌ **Архив** — историческая ценность

## 6. Правила соблюдения при работе с Базой

### 6.1. Технические правила

| Правило | Описание |
|---------|----------|
| **Неизменяемость оригинала** | Base не трогать → тесты в _TEST_ENV |
| **Контрольные суммы** | Git проверяет целостность |
| **Резервное копирование** | Git + _BACKUP + _TEST_ENV |
| **Актуальность** | Устаревшее → в _archive/ |

### 6.2. Организационные правила

| Правило | Описание |
|---------|----------|
| **Единообразие именования** | snake_case, латиница |
| **Документирование** | Оглавление в каждом файле |
| **Журнал изменений** | `reports/OPERATION_LOG.md` |
| **Срок хранения черновиков** | 7 дней (_drafts/) |

## 9. Пример рабочего процесса: добавление нового правила

### Шаг 1: Создание черновика

```powershell
# Создать черновик
New-Item "_drafts/new_naming_rule.md" -ItemType File
```

### Шаг 2: Написание правила

```markdown
# Правило именования файлов

**Статус:** ⏳ Черновик

## Описание

...
```

### Шаг 3: Анализ (7 дней)

```
[ ] Прочитать черновик
[ ] Оценить применимость
[ ] Проверить на дубликаты
[ ] Решить: сохранить / отложить / удалить
```

### Шаг 4: Размещение

```powershell
# Переместить в KNOWLEDGE_BASE
Move-Item "_drafts/new_naming_rule.md" "03-Resources/Knowledge/01_RULES/file_naming_rule.md"
```

### Шаг 5: Документирование

```powershell
# Записать в OPERATION_LOG.md
Add-Content "reports/OPERATION_LOG.md" "## 2026-03-02 Добавлено новое правило"
```

### Шаг 6: Git коммит

```powershell
git add 03-Resources/Knowledge/01_RULES/file_naming_rule.md
git commit -m "Add: правило именования файлов"
```
