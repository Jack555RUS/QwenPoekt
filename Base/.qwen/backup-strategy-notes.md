## 🎯 ПРАВИЛО 3-2-1 (ЗОЛОТОЙ СТАНДАРТ)

```
3 копии данных
2 разных типа носителей
1 копия вне офиса (offsite)
```

**Пример для нашей системы:**
```
3 копии:
  1. Оригинал: D:\QwenPoekt\Base\
  2. Локальный бэкап: D:\QwenPoekt\_BACKUP\
  3. Внешний бэкап: GitHub / USB диск / Облако

2 носителя:
  1. HDD (внутренний диск)
  2. USB диск / Облако (GitHub)

1 вне офиса:
  - GitHub (удалённо)
```

**Источник:** Datto, Veeam, NovaBACKUP (2026)

## 📁 ТИПЫ РЕЗЕРВНЫХ КОПИЙ

### **1. Полная копия (Full Backup)**

**Что:** Копируются **ВСЕ** данные

**Плюсы:**
- ✅ Надёжно (все данные в одном месте)
- ✅ Быстрое восстановление (один файл)

**Минусы:**
- ❌ Долго создаётся
- ❌ Много места

**Когда использовать:**
- Еженедельно/ежемесячно
- Перед крупными изменениями

---

### **2. Инкрементальная (Incremental)**

**Что:** Копируются изменения **с последней备份 любой типа**

**Пример:**
```
Пн: Full (100 GB)
Вт: Incremental (1 GB) — изменения с Пн
Ср: Incremental (500 MB) — изменения с Вт
Чт: Incremental (2 GB) — изменения с Ср
```

**Плюсы:**
- ✅ Быстро создаётся
- ✅ Мало места

**Минусы:**
- ❌ Долгое восстановление (нужна вся цепочка)

**Когда использовать:**
- Ежедневно/ежечасно
- При ограничении места

---

### **3. Дифференциальная (Differential)**

**Что:** Копируются изменения **с последнего Full**

**Пример:**
```
Пн: Full (100 GB)
Вт: Differential (1 GB) — изменения с Пн
Ср: Differential (1.5 GB) — изменения с Пн (включая Вт)
Чт: Differential (3.5 GB) — изменения с Пн (включая Ср)
```

**Плюсы:**
- ✅ Быстрее восстановление (нужен только Full + последний Differential)
- ✅ Меньше места чем Full

**Минусы:**
- ❌ Медленнее создания чем Incremental
- ❌ Больше места чем Incremental

**Когда использовать:**
- Баланс между скоростью создания и восстановления

## 🔄 СТРАТЕГИИ СИНХРОНИЗАЦИИ

### **1. Update (Обновление)**

**Поведение:**
```
Источник → Назначение
- Новые файлы → копируются
- Изменённые → обновляются
- Удалённые в источнике → СОХРАНЯЮТСЯ в назначении
```

**Когда:** Резервная копия с историей

---

### **2. Mirror (Зеркало)**

**Поведение:**
```
Источник = Назначение
- Новые файлы → копируются
- Изменённые → обновляются
- Удалённые в источнике → УДАЛЯЮТСЯ в назначении
```

**Когда:** Точная копия (осторожно!)

---

### **3. Bidirectional (Двусторонняя)**

**Поведение:**
```
Источник ↔ Назначение
- Изменения в обе стороны
- Конфликты → ручное решение
```

**Когда:** Синхронизация двух рабочих папок

## 🗑️ ОБРАБОТКА УДАЛЁННЫХ ФАЙЛОВ

### **Вариант A: Сохранить (для истории)**

```
Удалено в источнике → Сохранить в бэкапе
```

**Плюсы:**
- ✅ История сохраняется
- ✅ Можно восстановить случайно удалённое

**Минусы:**
- ❌ Бэкап растёт бесконечно

---

### **Вариант B: Удалить (зеркало)**

```
Удалено в источнике → Удалить в бэкапе
```

**Плюсы:**
- ✅ Бэкап = точная копия
- ✅ Не растёт размер

**Минусы:**
- ❌ Нельзя восстановить удалённое

---

### **Вариант C: Версионность (лучшее)**

```
Удалено в источнике → Переместить в $Recycle.Bin бэкапа
```

**Плюсы:**
- ✅ История сохраняется
- ✅ Бэкап не растёт бесконечно

**Минусы:**
- ⚠️ Сложнее реализация

## 📅 ПОЛИТИКИ ХРАНЕНИЯ (RETENTION)

### **1. По времени**

```
Хранить:
- Последние 24 часа (ежечасно)
- Последние 7 дней (ежедневно)
- Последние 30 дней (еженедельно)
- Последние 90 дней (ежемесячно)
```

**Удаление:**
```
Если файл старше 90 дней → удалить
```

---

### **2. По количеству версий**

```
Хранить:
- Последние 10 версий каждого файла
```

**Удаление:**
```
Если версий > 10 → удалить самые старые
```

---

### **3. По размеру бэкапа**

```
Максимальный размер: 100 GB
```

**Удаление:**
```
Если > 100 GB → удалить самые старые бэкапы
```

## 🏷️ ИМЕНОВАНИЕ И СТРУКТУРА

### **Формат имён папок:**

```
Base_YYYY-MM-DD_HH-mm
Base_2026-03-03_10-00

Где:
- YYYY-MM-DD = дата
- HH-mm = время
```

**Преимущества:**
- ✅ Сортировка по дате (естественный порядок)
- ✅ Читаемо человеком
- ✅ Нет проблем с символами

---

### **Структура бэкапа:**

```
_BACKUP/
├── Auto_Full/                    # Автоматические полные копии
│   ├── Base_2026-03-03_10-00/   # Папка с датой
│   ├── Base_2026-03-03_11-00/
│   ├── Base_2026-03-03_12-00/
│   └── backup.log               # Лог операций
│
├── Pre_Change/                   # Бэкап перед изменениями
│   ├── file_2026-03-03_10-30-00.ps1
│   └── backup.log
│
└── Manual/                       # Ручные бэкапы
    └── ...
```

## 🔄 ИНТЕГРАЦИЯ С GIT

### **Git + Бэкап:**

```
Git:
- ✅ Версионирование кода
- ✅ История изменений
- ✅ Ветвление
- ❌ Не хранит бинарные файлы (картинки, EXE)
- ❌ Не для больших объёмов

Бэкап:
- ✅ Полная копия (всё)
- ✅ Бинарные файлы
- ✅ Быстрое восстановление
- ❌ Нет версионирования (только по дате)

Вместе:
- Git → код, конфиги
- Бэкап → всё остальное
```

## 🛠️ ROBOCOPY — КЛЮЧИ И ПРИЁМЫ

### **Базовая команда:**

```cmd
robocopy <источник> <назначение> [ключи]
```

### **Ключи для бэкапа:**

| Ключ | Описание |
|------|----------|
| `/MIR` | Зеркалирование (Full) |
| `/Z` | Режим перезапуска при обрыве |
| `/R:n` | N попыток при ошибке (по умолчанию 1000000) |
| `/W:n` | N секунд между попытками (по умолчанию 30) |
| `/MT:n` | N потоков (1-128, по умолчанию 8) |
| `/NP` | Без прогресса (тихо) |
| `/NFL` | Без имён файлов |
| `/NDL` | Без имён каталогов |
| `/NS` | Без размеров |
| `/NC` | Без классов |
| `/NJS` | Без сводки |
| `/NJH` | Без заголовка |
| `/XD` | Исключить папки |
| `/XF` | Исключить файлы |
| `/LOG+:file` | Лог (дописывать) |

### **Пример полной команды:**

```cmd
robocopy D:\QwenPoekt\Base D:\QwenPoekt\_BACKUP\Auto_Full\Base_2026-03-03 ^
  /MIR /Z /R:2 /W:5 /MT:8 ^
  /NP /NFL /NDL /NS /NC /NJS /NJH ^
  /XD _TEMP sessions _LOCAL_ARCHIVE _BACKUP ^
  /XF *.tmp *.log *.cache ^
  /LOG+:D:\QwenPoekt\_BACKUP\backup.log
```

## 📊 СРАВНЕНИЕ: FULL VS INCREMENTAL VS DIFFERENTIAL

| Критерий | Full | Incremental | Differential |
|----------|------|-------------|--------------|
| **Время создания** | 🔴 Долго | 🟢 Быстро | 🟡 Средне |
| **Место на диске** | 🔴 Много | 🟢 Мало | 🟡 Средне |
| **Время восстановления** | 🟢 Быстро | 🔴 Долго | 🟡 Средне |
| **Надёжность** | 🟢 Высокая | 🟡 Средняя | 🟡 Средняя |
| **Сложность** | 🟢 Простая | 🟡 Средняя | 🟡 Средняя |

**Рекомендация (Macrium, 2026):**
- **Full:** Еженедельно
- **Incremental:** Ежедневно
- **Differential:** Для критичных данных

## 🎯 НАША СТРАТЕГИЯ (ПЛАН)

### **Уровень 1: Переписка ИИ (2 минуты)**

```
Что: sessions/YYYY-MM-DD_HH-mm/
Где: D:\QwenPoekt\Base\sessions\
Как: VBScript + PowerShell (невидимо)
Когда: Каждые 2 минуты
Версионность: Каждая сессия отдельно
```

---

### **Уровень 2: Бэкап перед изменением**

```
Что: Изменяемые файлы
Где: D:\QwenPoekt\_BACKUP\Pre_Change\
Как: pre-change-backup.ps1
Когда: Перед каждым изменением
Версионность: По timestamp
```

---

### **Уровень 3: Полная копия Base (10 минут)**

```
Что: Все файлы Base/ (с исключениями)
Где: D:\QwenPoekt\_BACKUP\Auto_Full\
Как: robocopy /MIR
Когда: Каждые 10 минут
Версионность: По дате (Base_YYYY-MM-DD_HH-mm)
```

**Исключения:**
- `_TEMP/` — временные файлы
- `sessions/` — и так сохраняются
- `_LOCAL_ARCHIVE/` — отдельный архив
- `_BACKUP/` — не копировать себя
- `PROJECTS/` — отдельный проект
- `OLD/` — архив
- `RELEASE/` — готовые проекты
- `KNOWLEDGE_BASE/` — большая, редко меняется
- `*.tmp`, `*.log`, `*.cache` — мусор

---

### **Уровень 4: GitHub (ежедневно)**

```
Что: Важные файлы (код, конфиги, правила)
Где: GitHub (origin/master)
Как: git push
Когда: В конце сессии / ежедневно
Версионность: Git commits
```

---

### **Уровень 5: Внешний диск (еженедельно)**

```
Что: Полная копия D:\QwenPoekt\
Где: USB диск / внешний HDD
Как: robocopy /MIR
Когда: Еженедельно
Версионность: По дате
```
