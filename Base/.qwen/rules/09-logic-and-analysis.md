# 🧠 ЛОГИКА И АНАЛИЗ (ПЕРЕД КАЖДЫМ ДЕЙСТВИЕМ)

**Версия:** 1.0

## 📋 ШАБЛОН ВОПРОСОВ (10 уровней)

### **УРОВЕНЬ 1: ЧТО? (Идентификация)**

```
1.1. Что именно мы делаем?
     - Тип действия (создать/изменить/удалить/проверить)
     - Конкретный объект (файл/папка/скрипт/правило)
     - Ожидаемый результат

1.2. Это уже существует?
     - Поиск аналогичных решений
     - Можно ли расширить существующее?
     - Изобретаем велосипед?
```

---

### **УРОВЕНЬ 2: ЗАЧЕМ? (Обоснование)**

```
2.1. Какую проблему решаем?
     - Симптом или корневая причина?
     - Действительно ли нужно решать?
     - Что случится, если НЕ решать?

2.2. Как это связано с текущими задачами?
     - Приоритет (срочно/важно/можно отложить)
     - Соответствует ли плану?
     - Не отклоняемся ли от цели?
```

---

### **УРОВЕНЬ 3: ГДЕ? (Расположение)**

```
3.1. Где это будет храниться?
     - Соответствует ли структуре PARA?
     - Правильная ли категория (Projects/Areas/Resources/Archives)?
     - Не нарушим ли существующую организацию?

3.2. Как это найти при перезагрузке?
     - Записать в CURRENT_CONTEXT.md?
     - Записать в SESSION_DIARY.md?
     - Добавить в реестр (если есть)?
```

---

### **УРОВЕНЬ 4: СВЯЗИ? (Интеграция)**

```
4.1. Какие связи с другими файлами?
     - Какие файлы ссылаются на это?
     - На какие файлы это ссылается?
     - Нужно ли обновлять ссылки?

4.2. Как это влияет на систему?
     - Зависимости (что зависит от этого?)
     - Последствия (что изменится?)
     - Совместимость (работает ли с существующим?)
```

---

### **УРОВЕНЬ 5: ТЕСТЫ? (Проверка)**

```
5.1. Нужны ли тесты?
     - Критично ли это для системы?
     - Можно ли протестировать вручную?
     - Стоит ли автоматизировать?

5.2. Где тестировать?
     - Docker (полная изоляция)
     - _TEST_ENV/ (копия Base)
     - Параллельный workspace
     - Продакшен (только если безопасно)
```

---

### **УРОВЕНЬ 6: БЕЗОПАСНОСТЬ? (Риски)**

```
6.1. Что может сломаться?
     - Какие файлы затронуты?
     - Какие процессы нарушатся?
     - Есть ли резервная копия?

6.2. Как откатить?
     - Git restore?
     - Резервная копия?
     - Ручной откат?
     - Время на откат?
```

---

### **УРОВЕНЬ 7: ДОКУМЕНТАЦИЯ? (Фиксация)**

```
7.1. Что задокументировать?
     - Создание файла
     - Изменение правила
     - Новую функцию
     - Тесты и результаты

7.2. Где задокументировать?
     - CURRENT_CONTEXT.md (текущая сессия)
     - SESSION_DIARY.md (долгосрочная память)
     - README.md (реестр)
     - Commit message (Git)
```

---

### **УРОВЕНЬ 8: ПРИОРИТЕТЫ? (Порядок)**

```
8.1. Что делать сначала?
     - Зависимости (что от чего зависит)
     - Критичность (что важнее)
     - Время (что можно отложить)

8.2. Что можно параллельно?
     - Независимые задачи
     - Автоматизация
     - Делегирование
```

---

### **УРОВЕНЬ 9: РЕСУРСЫ? (Затраты)**

```
9.1. Сколько времени?
     - Создание
     - Тестирование
     - Документирование
     - Поддержка

9.2. Сколько места?
     - Размер файла
     - Резервные копии
     - Тестовые среды
```

---

### **УРОВЕНЬ 10: ИТОГ? (Решение)**

```
10.1. Продолжаем или останавливаем?
      - Все вопросы отвечены?
      - Риски приемлемы?
      - Ресурсы есть?

10.2. Кто принимает решение?
      - ИИ (автономно)
      - Пользователь (подтверждение)
      - Вместе (обсуждение)
```
